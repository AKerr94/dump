Continuous Deployment Continued: CodeDeploy problem and workaround

Our initial plan was to use CodeDeploy, but after setting up the roles and profiles in AWS we found out that the Amazon accounts we had been provided with did not have access to the service. This was a big problem for us as CodeDeploy was a huge part of the Continuous Delivery pipeline and using it would have meant successful builds would swiftly and automatically be rolled out across our instances.

Without this tool at our disposal we needed to find an alternative way. At this point we had fully implemented Continuous Integration with Jenkins for the Tomcat application, and so needed to work out how to get this build sent to our servers. We installed several Jenkins plugins to handle the post build steps; PostBuildScript allows for bash scripts to be executed as a post build action, and Artifact deployer allows for archived artifacts to be relocated to a more manageable location.

The specific implementation copies the archived build artifact (in the case of this Tomcat application, .war file) to a folder /builds/ owned by the Jenkins user, and then executes a simple scp transfer of the artifact to the Puppet master server (this involved configuring a limited permissions Jenkin user on the Puppet server with the public key from the Jenkins machine pasted into the authorized_keys for this new user). This meant that the Jenkins portion of the pipeline for this application was complete, with a push to the GitHub repository initiating a Jenkins build which, if successful, automatically sends the .war file to the Puppet master server which will then be used to provision the agent nodes.

This is not quite as elegant a solution as originally envisioned, but is necessary considering the limitations faced with the fact that we are not able to use CodeDeploy. The main issue with this setup is the increased overhead which is incurred and the unavoidable delay in updating agent nodes; while with CodeDeploy instances would begin to be updated immediately, with Puppet these updates will only apply when the agents check in to the master. By default this is every 30 minutes, but reducing this window will mean shorter turn-around times for updates at the cost of increased overhead. Nonetheless, we have successfully configured the backbones of a Continuous Delivery pipeline, and now must just finish up the Puppet configurations and then move on to other tasks.

Continuous Integration with Jenkins and GitHub

A core part of our overall system design is the Continuous Delivery pipeline, the primary software tools used being Jenkins and Amazon Web Services CodeDeploy. The overall aim of implementing these technologies was to allow for Continuous Deployment through linking remote GitHub repositories to a Jenkins server which builds the code, tests it, and – assuming these tests pass – instigates CodeDeploy to deploy the new build across the relevant AWS instances.

Bilal and myself were responsible for this section of the project, and the first step in this process was to manually configure Jenkins. This helped us get a better understanding of how the software worked, and laid the foundations for a largely automated process later. Ubuntu 14.04 was the platform for this server, and setting up the CI platform necessitated installing Jenkins, git plugins, JDK7, maven and also apache2 (not necessary, but useful for enabling us to proxy Jenkins through to port 80 and using the public DNS, making the Jenkins interface accessible through the world wide web with a domain name assigned by Amazon). Lots of configurations needed to be made through the Jenkins interface, for example configuring the MAVEN HOME location so that the developer’s Java app could be built with Jenkins. Importantly, considering the globally accessible nature of this server, security configurations were made so that anonymous users have read only access and an administrator login allows for full privileges.

A crucial requirement for facilitating this build was to link the relevant GitHub repository to the Jenkins server. Firstly, an SSH key was generated for the ‘jenkins’ user on the Jenkins server, which was used as a Deploy Key for the specific repository, a measure that was implemented through the settings in GitHub. This authentication was confirmed through the command line on the Jenkins server, as shown below.

With the Deploy Key successfully associated, the next measure we took was to setup a webhook. This means that an action on GitHub (such as a push to the repository) triggers an action, sending this information to the Jenkins server so that it can act appropriately (in our case we did indeed want Jenkins to automatically trigger a build following a push to the linked repository). This had to be setup in the repository’s settings with special links that Jenkins could handle, and on the Jenkins side the GitHub repository had to be linked along with authorization tokens for the associated account. Linking Jenkins and GitHub like this is a little more complex than other methods but doing so in this way means that Jenkins can build from a private repository, which is beneficial for example if developers do not want their code to be made public. Successful builds will need to trigger AWS CodeDeploy to deploy the application across our instances, and failed builds are logged, with monitoring available through the Jenkins interface and dashboards giving an overview of build health and allowing for an audit trail that can be used for error troubleshooting.

In the case of a build failure Jenkins can also be configured to notify administrators or specific email accounts (such as the developer whose build is broken!). We tested this functionality with an email address for our team.

We tested that this configuration was working properly, as shown below.

With the Continuous Integration portion of the setup working, we then worked on automating the process. After initial issues with Puppet configurations, and, considering that we were already using the advanced options of AWS to provision Jenkins on our initial test servers with bash scripts, we decided to extend these with a more involved and all-round script that provisioned a Jenkins server pre-configured with many of the necessary plugins and setup options, for example dynamically inputting the initiated instances public DNS into the Apache configuration in order to allow for access over the web through the Amazon hostname (otherwise this would simply lead to the Apache default splashpage on port 80 rather than redirecting to the Jenkins interface).

With this bash script a Jenkins server can be provisioned extremely quickly with AWS, and serves as a possible route for backup/ disaster recovery in the event of a server failure, although our main disaster recovery mechanism for the Jenkins server comes in the form of an AMI created from this final server configuration, which can be deployed as an AWS instance as needed. Therefore, this AMI would be our first port of call in the event of server failure, but with the bash script we have redundancy, so, for example, were AWS to fail, or, should we want to switch to a different service, we would not be reliant on these services and could provision another server with another provider.

The next steps we will take will be to setup CodeDeploy and connect this to Jenkins, so that successful builds will be automatically deployed across our infrastructure. In this manner, we will have achieved Continuous Deployment as pushes to the GitHub repository will initiate Jenkins builds that, if successful, will update existing instances with the new code/ applications.

Setting up a Puppet module to automate activities

In this documentation I will discuss how I created a puppet module to automate the following activities :

– install apache and PHP

– Install MySQL

– Create a database and a database user on MySQL for WordPress

– Install and configure WordPress

In order to perform these tasks I needed to spin up a ubuntu 14.04 virtual machine through AWS as well as having access to a sudo user.

first step was to get the package downloaded using this command into the terminal via the ubuntu machine

wget https://apt.puppetlabs.com/puppetlabs-release-trusty.deb
sudo dpkg -i puppetlabs-release-trusty.deb

once this file had been downloaded I could then install puppet as well as checking the version which I had downloaded using these commands.

sudo apt-get update
sudo apt-get install puppet

sudo puppet --version

The next stage was to install the puppet lab apache and mysql modules using the commands shown below.

sudo puppet module install puppetlabs-apache
sudo puppet module install puppetlabs-mysql

once I had completed this step I could then list the modules installed on the ubuntu machine as shown in the diagram below.

Upon the installation of the packages I then had to create a new directory in which I would store/create the modules. (mkdir MyModules)

puppet module generate do-wordpress --skip-interview

vi ~/MyModules/do-wordpress/metadata.json

I then generated the modules and edited the metadata.json tag into the text editor, this has to be done because of a bug in puppet.

The next step was to start creating the manifest which would install Apache and PHP, I created the file through the “vi” text editor

vi ~/MyModules/do-wordpress/manifests/web.pp

class wordpress::web {

    # Install Apache
    class {'apache': 
        mpm_module => 'prefork'
    }

    # Add support for PHP 
    class {'::apache::mod::php': }
}

in the screenshot above you can see that I added the classes as well as using a prefork via the multi-processing-module to increase compatibility with other modules.

The next step was to create a file to store configuration variables

vi ~/MyModules/do-wordpress/manifests/conf.pp

class wordpress::conf {
    # You can change the values of these variables
    # according to your preferences

    $root_password = '*********'
    $db_name = 'wordpress'
    $db_user = 'wp'
    $db_user_password = '*******'
    $db_host = 'localhost'

    # Don't change the following variables

    # This will evaluate to wp@localhost
    $db_user_host = "${db_user}@${db_host}"

    # This will evaluate to wp@localhost/wordpress.*
    $db_user_host_db = "${db_user}@${db_host}/${db_name}.*"
}

Above is the screenshot showing some of my test configuration.

The next stop was to create the manifest for MySQL, this manifest would perform the following operations;

– Install the MySQL server

– Sets the root password for MySQL Server

– Creates a database for WordPress

– Creates a user for WordPress

– Grants privileges to the user to access the database

– Install the MySQL client and bindings for various languages.

class wordpress::db {

    class { '::mysql::server':

        # Set the root password
        root_password => $wordpress::conf::root_password,

        # Create the database
        databases => {
            "${wordpress::conf::db_name}" => {
                ensure => 'present',
                charset => 'utf8'
            }
        },

        # Create the user
        users => {
            "${wordpress::conf::db_user_host}" => {
                ensure => present,
                password_hash => mysql_password("${wordpress::conf::db_user_password}")
            }
        },

        # Grant privileges to the user
        grants => {
            "${wordpress::conf::db_user_host_db}" => {
                ensure     => 'present',
                options    => ['GRANT'],
                privileges => ['ALL'],
                table      => "${wordpress::conf::db_name}.*",
                user       => "${wordpress::conf::db_user_host}",
            }
        },
    }

    # Install MySQL client and all bindings
    class { '::mysql::client':
        require => Class['::mysql::server'],
        bindings_enable => true
    }
}

All of the operations are performed through the classes ::mysql::server & ::mysql::client

The following step is simply to download the latest version of WordPress, in order to do this I had to create a new file in the directory “MyModules” this file was to be called files

wget http://wordpress.org/latest.tar.gz

above is the command to download the latest WordPress.

The next step was to create a template for the wp-config.php file, again I created a new file which I added to the do-wordpress folder, this file is named templates. I then moved in the /tmp folder and extracted the latest WordPress files.

tar -xvzf ~/MyModules/do-wordpress/files/latest.tar.gz  # Extract the tar

cp /tmp/wordpress/wp-config-sample.php ~/MyModules/do-wordpress/templates/wp-config.php.erb

I then cleaned up the /tmp folder

rm -rf /tmp/wordpress  # Clean up

vi ~/MyModules/do-wordpress/templates/wp-config.php.erb

I then edited the wp-config.php.erb file.

<?php
define('DB_NAME', '<%= scope.lookupvar('wordpress::conf::db_name') %>');
define('DB_USER', '<%= scope.lookupvar('wordpress::conf::db_user') %>');
define('DB_PASSWORD', '<%= scope.lookupvar('wordpress::conf::db_user_password') %>');
define('DB_HOST', '<%= scope.lookupvar('wordpress::conf::db_host') %>');
define('DB_CHARSET', 'utf8');
define('DB_COLLATE', '');

define('AUTH_KEY',         'put your unique phrase here');
define('SECURE_AUTH_KEY',  'put your unique phrase here');
define('LOGGED_IN_KEY',    'put your unique phrase here');
define('NONCE_KEY',        'put your unique phrase here');
define('AUTH_SALT',        'put your unique phrase here');
define('SECURE_AUTH_SALT', 'put your unique phrase here');
define('LOGGED_IN_SALT',   'put your unique phrase here');
define('NONCE_SALT',       'put your unique phrase here');

$table_prefix  = 'wp_';

define('WP_DEBUG', false);

if ( !defined('ABSPATH') )
    define('ABSPATH', dirname(__FILE__) . '/');

require_once(ABSPATH . 'wp-settings.php');

the next step was to create the manifest for WordPress

vi ~/MyModules/do-wordpress/manifests/wp.pp

I created this manifest to perform the following operations;

– copy the contents of the WordPress installation bundle to /var/www/ – this has to be done as apache has a default configuration which serves files to /var/www/

– Generates the wp-config.php file using a template.

class wordpress::wp {

    # Copy the WordPress bundle to /tmp
    file { '/tmp/latest.tar.gz':
        ensure => present,
        source => "puppet:///modules/wordpress/latest.tar.gz"
    }

    # Extract the WordPress bundle
    exec { 'extract':
        cwd => "/tmp",
        command => "tar -xvzf latest.tar.gz",
        require => File['/tmp/latest.tar.gz'],
        path => ['/bin'],
    }

    # Copy to /var/www/
    exec { 'copy':
        command => "cp -r /tmp/wordpress/* /var/www/",
        require => Exec['extract'],
        path => ['/bin'],
    }

    # Generate the wp-config.php file using the template
    file { '/var/www/wp-config.php':
        ensure => present,
        require => Exec['copy'],
        content => template("wordpress/wp-config.php.erb")
    }
}

the next step was to create a .init.pp file, a manifest that integrates the other manifests.

vi ~/MyModules/do-wordpress/manifests/init.pp

this is the code to enter into the text editor to alter the init.pp file.

class wordpress {
    # Load all variables
    class { 'wordpress::conf': }

    # Install Apache and PHP
    class { 'wordpress::web': }

    # Install MySQL
    class { 'wordpress::db': }

    # Run WordPress installation only after Apache is installed
    class { 'wordpress::wp': 
        require => Notify['Apache Installation Complete']
    }

    # Display this message after MySQL installation is complete
    notify { 'MySQL Installation Complete':
        require => Class['wordpress::db']
    }

    # Display this message after Apache installation is complete
    notify { 'Apache Installation Complete':
        require => Class['wordpress::web']
    }

    # Display this message after WordPress installation is complete
    notify { 'Wordpress Installation Complete':
        require => Class['wordpress::wp']
    }
}

You can see comments (#) about what each operation performs in the screenshot above.

The next step was to build the WordPress module, this initial task i did was to change directory back to the “MyModules” folder, from here I then performed the following command.

sudo puppet module build do-wordpress

this command built the WordPress module. when entering this command I received the following message.

Notice: Building /home/user/MyModules/do-wordpress for release
Module built: /home/user/MyModules/do-wordpress/pkg/do-wordpress-0.1.0.tar.gz

Once I had built the WordPress module I then began to install it using the following command.

sudo puppet module install ~/MyModules/do-wordpress/pkg/do-wordpress-0.1.0.tar.gz

To test that the everything worked correctly I decided to use the module in a standalone manifest file to install WordPress, to do this I entered the text editor for the appropriate file.

vi /tmp/install-wp.pp

I made the correct alterations to the file and then ran a puppet apply command.

class { 'wordpress':
}
sudo puppet apply /tmp/install-wp.pp

Once this was complete I then ran the IP address into the URL which then gave me access to the WordPress service which we now use to document each task that we have done during the course of the DevOps case study 2015.

The final stage was to then add my team as users so that they could also document what they had done as part of the team, I assigned them all administrative roles so that they had full access to whatever they needed.
